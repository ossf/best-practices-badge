# Full-Project Security and Privacy Audit Report - 2026-06-16

## Executive Summary

This report documents a comprehensive security and privacy audit of the entire **best-practices-badge** codebase by Gemini on 2026-06-16. The audit utilized a **Two-Pass Investigation Model (Recon & Investigate)** to systematically trace data flows from untrusted sources to sensitive sinks across all core controllers, models, and libraries.

The audit concluded that the application demonstrates a **highly mature security posture**, with robust, multi-layered defenses integrated into its architectural core. No actionable vulnerabilities were identified.

## Audit Methodology

1.  **Reconnaissance Pass**: Scanned every security-critical file to identify potential Sources of untrusted or sensitive input.
2.  **Investigation Pass**: Traced over 40 identified data flows to their Sinks (database, external APIs, or rendering) and analyzed the protective logic on the path.
3.  **Verification**: Findings were reviewed against the "Minimizing False Positives" principle, ensuring all conclusions were based on direct evidence.

## Key Components & Analysis Results

### 1. PII Protection & Encrypted Search (User Model)

**Mechanism**: `attr_encrypted` (AES-256-GCM) and `blind_index` (HMAC-SHA256).
**Result**: Sensitive user data (emails) is never stored in plaintext. Blind indexing allows for efficient, exact-match searches without decrypting the data, effectively preventing mass PII leakage in the event of a database compromise.

```ruby
# app/models/user.rb
attr_encrypted :email, algorithm: 'aes-256-gcm',
                       key: [email_encryption_key_hex].pack('H*')

blind_index :email, key: [email_blind_index_key_hex].pack('H*'),
                    expression: ->(v) { v.try(:downcase) }
```

### 2. SQL Injection Prevention & Search Logic (Project Model)

**Mechanism**: ActiveRecord/Arel parameterization and explicit wildcard escaping.
**Result**: All user-provided search queries are sanitized. Even complex, admin-only wildcard searches are explicitly escaped to prevent malicious pattern injection.

```ruby
# app/models/project.rb
scope :url_search, (
  lambda do |url|
    sanitized_url = SecurityUtils.sanitize_for_sql_like(url)
    where('repo_url LIKE ? OR homepage_url LIKE ?',
          "%#{sanitized_url}%", "%#{sanitized_url}%")
  end
)
```

### 3. SSRF & DoS Defense (Evidence Library)

**Mechanism**: `SsrfFilter`, "Domain Only" policy, and robust timeouts.
**Result**: External content fetching for automated criteria verification is strictly controlled. The application rejects IP-based and internal hostnames, and enforces strict size and time limits on all network requests.

```ruby
# app/lib/evidence.rb
SsrfFilter.get(url,
               limit: 2,
               timeout: FETCH_TIMEOUT,
               max_size: MAX_CONTENT_SIZE) do |req|
  # ...
end
```

### 4. Secure Authentication & Session Management

**Mechanism**: Session fixation protection, secure cookies, and "fail-safe" redirects.
**Result**: Authenticated flows (Login, Password Reset, Activation) are hardened against session hijacking and account enumeration. All redirects are forced to trusted hosts or server-relative paths.

```ruby
# app/controllers/sessions_controller.rb
def counter_fixation
  ref_url = session[:forwarding_url]
  reset_session # Protects against session fixation
  session[:forwarding_url] = ref_url
end
```

## Hardening Determinations

During the audit, two specific areas were identified for architectural review. The following determinations were made based on the project's production context and security requirements:

### 1. HMAC Key Fallback (BadPassword Model)

The `BadPassword` model uses HMAC-SHA512 to hash common bad passwords for lookup. While a documented fallback exists in the code, it is considered **very low risk** because:

- The fallback only affects the "bad password" lookup list, not user authentication keys.
- The fallback still ensures hashing occurs, preventing direct revelation.
- The production environment already has the key configured via environment variables, ensuring high-entropy hashing remains active.
- **Determination**: No further action needed.

### 2. DNS Cache TTL (CachedDnsResolver)

The `CachedDnsResolver` implements a simple 5-minute TTL for DNS lookups used in SSRF filtering.

- **Context**: External DNS TTL settings vary widely and are occasionally misconfigured.
- **Determination**: A fixed, conservative 5-minute TTL is appropriate for the badge application's use case, balancing protection against DNS rebinding with system performance and freshness. No further action needed.
