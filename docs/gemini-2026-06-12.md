# Security Audit Report - 2026-06-12

## Summary

I have conducted a security review of the project, focusing on common web vulnerabilities and the specific implementation details of this application. Overall, the project demonstrates a strong security posture with robust protections for authentication, data handling, and privacy.

## Findings

### 1. Privacy: Gravatar Integration (Verified Intentional)

- **Status**: Not a Vulnerability / User Preference
- **Location**: `app/models/user.rb`, `app/helpers/users_helper.rb`
- **Description**: The application uses Gravatar to display user avatars. This involves exposing the MD5 hash of the user's email address in the image URL.
- **Verification**: Investigation confirmed that the application only exposes the real MD5 hash if a Gravatar image actually exists for that email (verified via `gravatar_exists?`). If no Gravatar exists, a bogus hash is used, preventing the leak of hashes for non-Gravatar users.
- **Justification**: Users who have set up a Gravatar for their email address generally expect and consent to its use across services where they register with that email. Since the application respects this by only enabling it when an image is present, this is considered an intentional feature rather than a vulnerability.

### 2. Authorization: Editor Management (Verified Intentional)

- **Status**: Not a Vulnerability
- **Location**: `app/controllers/projects_controller.rb#update_additional_rights`
- **Description**: The application allows any user with "edit" permissions to add other editors to the project. While only "controllers" (owners/admins) can remove editors, the ability for editors to grant access to others was reviewed.
- **Justification**: This is confirmed as expected and desired behavior to facilitate collaboration. Once a user is trusted as an editor, they are also trusted to expand the collaboration team as needed.

### 3. Rate Limiting: Hardening against Misconfiguration and Spoofing

- **Status**: Targeted for Hardening
- **Location**: `app/lib/client_ip.rb`, `config/initializers/rack_attack.rb`, `config/application.rb`
- **Description**: The current `ClientIp.acquire` method uses a hardcoded index (`[-2]`) to extract the user's IP from the `X-Forwarded-For` header. This is brittle if the proxy chain length changes and is susceptible to IP spoofing if an attacker provides their own `X-Forwarded-For` header.

#### Options for Addressing:

| Option | Description | Pros | Cons |
| :--- | :--- | :--- | :--- |
| **1. Refined Manual Acquisition** | Add fallbacks and bounds checking to `ClientIp.acquire` (e.g., `ips[-2] \|\| ips.last`). | Simple, self-contained, no new dependencies. | Brittle, does not prevent spoofing, ignores industry standards. |
| **2. Rails `trusted_proxies`** | Use Rails' built-in `ActionDispatch::RemoteIp` logic by configuring `config.action_dispatch.trusted_proxies`. | **Selected.** Industry standard, secure against spoofing, leverages battle-tested framework code. | Requires maintaining a list of trusted IP ranges (e.g., CDN IPs). |
| **3. Env-Based Offset** | Use an environment variable (e.g., `X_FORWARDED_FOR_OFFSET`) to control the index. | Flexible across different environments. | Adds operational complexity, opaque logic, doesn't solve spoofing. |

#### Decision: Option 2 (Rails Standard `trusted_proxies` with Dynamic Loading)

We have selected **Option 2** with an added layer of resilience: **Dynamic Loading with Static Fallback**. This ensures the system stays up-to-date with Fastly's infrastructure changes while remaining reliable even during network outages.

#### Implementation Details & Infrastructure Analysis:

**1. Resilient TRUSTED_PROXIES Configuration**
To ensure reliable operation on Heroku with Fastly as the CDN, the application will attempt to dynamically fetch trusted IPs at boot time.
- **Dynamic URL**: `TRUSTED_PROXIES_URL` set to `https://api.fastly.com/public-ip-list`.
- **Static Fallback**: `TRUSTED_PROXIES` environment variable. For staging and production, this must be set to the following Fastly CIDR ranges to ensure the fallback is effective:
  ```text
  23.235.32.0/20,43.249.72.0/22,103.244.50.0/24,103.245.222.0/23,103.245.224.0/24,104.156.80.0/20,140.248.64.0/18,140.248.128.0/17,146.75.0.0/17,151.101.0.0/16,157.52.64.0/18,167.82.0.0/17,167.82.128.0/20,167.82.160.0/20,167.82.224.0/20,172.111.64.0/18,185.31.16.0/22,199.27.72.0/21,199.232.0.0/16,2a04:4e40::/32,2a04:4e42::/32
  ```
- **Disablement Option**: `TRUSTED_PROXIES_DISABLED` environment variable. If set to `true`, the application will bypass these checks (useful for internal/private deployments).
- **Fail-Fast**: In production, if both methods fail (no URL data and no static list) and it's not disabled, the application will raise a `SecurityAssertionError` at boot time.

**2. Detailed Method Design (`SecurityUtils`)**:
We will implement `SecurityUtils.load_trusted_proxies(url:, static:, fail_fast:)` in `lib/security_utils.rb`:
```ruby
def self.load_trusted_proxies(url: nil, static: nil, fail_fast: false)
  ips = []
  # Try dynamic URL first
  if url.present?
    begin
      response = HTTParty.get(url, timeout: 5)
      if response.success?
        # Fastly format: { 'addresses' => [...], 'ipv6_addresses' => [...] }
        json = response.parsed_response
        ips += (json['addresses'] || []) + (json['ipv6_addresses'] || [])
      end
    rescue StandardError => e
      Rails.logger.warn "Dynamic proxy load failed: #{e.message}. Falling back to static list."
    end
  end

  # Fallback to static list if URL failed or was empty
  if ips.empty? && static.present?
    ips = static.split(',').map(&:strip)
  end

  # Fail-fast check for production
  if ips.empty? && fail_fast && !ENV['TRUSTED_PROXIES_DISABLED'] == 'true'
    security_assertion(false, "No trusted proxies configured via URL or static list.")
  end

  ips.map { |ip| IPAddr.new(ip) }.freeze
end
```

**3. Documentation & Inline Comments**:
To prevent future maintainers from misusing this list, the following comment will be included in `lib/security_utils.rb`, `config/application.rb`, and `app/lib/client_ip.rb`:
> "The trusted proxies' IP addresses are *not* being used for user authentication; they're being used to counter CDN piercing and to ensure that our rate limits apply to the correct IP addresses."

**4. Boot-time Configuration**:
In an initializer (e.g., `config/initializers/fastly.rb`), we will call this method and assign the result to `config.action_dispatch.trusted_proxies`.

**5. Cloud Piercing (Bypassing Fastly)**
- **Argument For (Weakness)**: Attackers reaching the Heroku URL directly bypass Fastly's edge protections (WAF, edge-caching, and DDoS mitigation).
- **Argument Against (Not a Rate-Limit Weakness)**: The `ActionDispatch::RemoteIp` logic is robust. If a user hits Heroku directly, the Heroku router (trusted by default) is the immediate proxy; `RemoteIp` will see the direct client IP as the first untrusted source. Identification remains accurate.
- **Verdict**: While cloud piercing is a general **Edge Security Weakness**, it does **not** undermine the accuracy of the rate-limiting system once `trusted_proxies` is configured.

**6. Startup Middleware Order Verification**
To ensure that `ActionDispatch::RemoteIp` always processes the request before `Rack::Attack` (so that `req.remote_ip` is available), we will add a check in a new initializer:
```ruby
# config/initializers/rate_limiting_check.rb
remote_ip_idx = Rails.application.config.middleware.index(ActionDispatch::RemoteIp)
rack_attack_idx = Rails.application.config.middleware.index(Rack::Attack)
if remote_ip_idx.nil? || rack_attack_idx.nil? || remote_ip_idx > rack_attack_idx
  raise "Security Configuration Error: RemoteIp must precede Rack::Attack in middleware stack."
end
```

#### Testing Strategy (100% Statement Coverage):

To achieve 100% coverage, we will use a dedicated test file `test/unit/lib/security_utils_proxy_test.rb` and use `WebMock` (or simple stubs) to simulate:
- **Successful Dynamic Load**: Verify all IPs from Fastly's JSON format are parsed correctly.
- **Failed Dynamic Load / Timeout**: Verify it falls back to the `TRUSTED_PROXIES` static list.
- **Malformed JSON**: Verify it falls back to the static list.
- **Both Missing (Production)**: Verify `SecurityAssertionError` is raised when `fail_fast: true`.
- **Both Missing (Development)**: Verify it returns an empty array (defaulting to Rails internal trust) without crashing.
- **Invalid IP Formats**: Verify `IPAddr` exceptions are caught or handled during initialization.
- **Configuration Consistency**: A test to verify that the application raises an error in production if no proxies are configured, but boots normally in development.
- **Middleware Ordering**: Assert the correct position of `Rack::Attack` relative to `RemoteIp` in the middleware stack.

### 4. General Observations (Verified Safe)

- **SQL Injection**: Search logic in `ProjectsController` and `Project` model scopes is well-protected using Arel's parameterization and `sanitize_sql_like`.
- **XSS**: High-risk Markdown areas were previously audited as safe. The strict CSP policy (no `unsafe-inline`) provides a strong second layer of defense.
- **Open Redirects**: The application uses robust guards (`valid_return_path?`, `allow_other_host: false`) for redirections.
- **Mass Assignment**: Sensitive fields like `user_id` and `role` are protected by explicit checks and filtered `project_params`.

## Conclusion

The application follows modern security best practices. The identified items are primarily minor privacy or logic refinements rather than critical vulnerabilities.
