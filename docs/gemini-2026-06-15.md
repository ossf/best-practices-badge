# Security and Privacy Audit Report - 2026-06-15

## Executive Summary

This report documents a comprehensive security and privacy audit performed on the core infrastructure and security hardening changes introduced in the June 2026 pull request. The audit utilized a **Two-Pass Investigation Model (Recon & Investigate)** to systematically trace data flows from untrusted sources to sensitive sinks.

The audit concluded that the current changes are **robust and secure**, leveraging multi-layered, "fail-fast" defenses to protect the application's origin and ensure accurate client identification.

## Audit Methodology

1.  **Reconnaissance Pass**: Scanned all modified files to identify potential Sources of untrusted or sensitive input.
2.  **Investigation Pass**: Traced identified data flows to their Sinks (execution, rendering, or storage points) and analyzed the sanitization, validation, and escaping logic on the path.
3.  **Verification**: Findings were reviewed against the "Minimizing False Positives" principle, ensuring all conclusions were based on direct evidence and actionable code.

## Analyzed Components & Results

### 1. Origin Shielding & CDN Piercing Prevention

**Source**: `request.forwarded_for&.last` in `ApplicationController`.
**Sink**: `verify_origin_shielding` guard.

The audit verified that the application correctly identifies the "anchor" IP—the address that reached the Heroku router—from the end of the `X-Forwarded-For` chain. This IP is matched against a verified list of edge proxies to prevent attackers from bypassing the CDN.

```ruby
def verify_origin_shielding
  return unless ENFORCE_ORIGIN_SHIELDING
  last_proxy = request.forwarded_for&.last
  return if SecurityUtils.edge_proxy?(last_proxy)
  render plain: '403 Forbidden - Direct origin access not allowed', status: :forbidden
end
```

### 2. Secure Client IP Identification

**Source**: `X-Forwarded-For` and `X-Real-IP` headers.
**Sink**: Rate limiting (`Rack::Attack`) and client-side IP logging.

The application leverages Rails' hardened `ActionDispatch::RemoteIp` logic to traverse the proxy chain securely from right-to-left, stopping at the first non-trusted IP. This ensures rate limits are applied to the true client, not a proxy or spoofed header.

```ruby
def self.extract(req)
  return req.remote_ip if req.respond_to?(:remote_ip)
  ActionDispatch::Request.new(req.env).remote_ip
end
```

### 3. SSRF Protection & "Domain Only" Policy

**Source**: User-submitted project and homepage URLs.
**Sink**: `Evidence` collection and database storage.

The `dubious_url?` utility enforces a strict policy rejecting IP-based hostnames and internal hostnames (e.g., `localhost`). This provides robust protection against Server-Side Request Forgery (SSRF) during evidence gathering.

```ruby
def self.dubious_url?(url)
  # ... (URI parsing)
  host = uri.host
  return true if host.exclude?('.') # Blocks localhost, internal hostnames
  begin
    IPAddr.new(host)
    return true # Rejects literal IP addresses
  rescue IPAddr::InvalidAddressError
    return true if host.match?(/\A(0x[0-9a-f]+|[0-9.]+)\z/i) # Rejects hex/integer bypasses
  end
  # ...
end
```

### 4. Middleware Integrity & Boot-Time Assertions

**Control**: Boot-time check of the middleware stack.

The audit confirmed the implementation of "fail-fast" security assertions that prevent the application from starting if critical security invariants are violated, such as improper middleware ordering.

```ruby
Rails.application.config.after_initialize do
  middleware_classes = Rails.application.config.middleware.map(&:klass)
  remote_ip_idx = middleware_classes.index(ActionDispatch::RemoteIp)
  rack_attack_idx = middleware_classes.index(Rack::Attack)
  if remote_ip_idx.nil? || rack_attack_idx.nil? || remote_ip_idx > rack_attack_idx
    SecurityUtils.security_assertion(false, 'RemoteIp must precede Rack::Attack.')
  end
end
```

### 5. Open Redirect Prevention

**Source**: `request.original_url` and `session[:forwarding_url]`.
**Sink**: `redirect_to`.

Locale and session redirections are hardened by forcing the host to a trusted `PUBLIC_HOSTNAME` or validating that the path is server-relative (starting with `/` but not `//`).

```ruby
def self.safe_localized_internal_url(original_url, locale)
  url = URI.parse(original_url)
  url.host = ENV.fetch('PUBLIC_HOSTNAME', url.host)
  # ... (path reconstruction with locale)
  url.to_s
end
```
